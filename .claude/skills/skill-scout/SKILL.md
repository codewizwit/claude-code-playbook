# Skill Scout

Analyze this codebase and the current session's workflow for repetitive patterns that could be automated with Claude Code skills. Output a prioritized list of skill recommendations with evidence and draft SKILL.md skeletons.

This skill can be invoked manually via `/skill-scout` or triggered automatically when the session context is compacted (a sign the session has been long and active enough to have real patterns worth capturing).

## Analysis Steps

### 1. Analyze Session Workflow

Before scanning the codebase, look at what happened in this session:

- Review the conversation history for repeated multi-step tasks — sequences of commands or tool calls that followed the same pattern more than once.
- Identify recurring prompt patterns — similar requests the user made multiple times (e.g., "generate a test for...", "add a migration for...", "review this PR...").
- Note any manual coordination steps — places where the user had to explain the same context, link the same files, or repeat instructions that could have been captured as a skill.
- Check for repeated command sequences in Bash history — chains of commands that always run together.

### 2. Map the Codebase

- Use `Glob` to catalog the directory structure, file types, and naming conventions.
- Identify co-located file groups (e.g., component + test + style + story files that always appear together).
- Note any generator configs, template directories, or scaffolding scripts already in place.

### 3. Detect Repetitive Code Patterns

Scan for these categories of automation opportunity:

**Boilerplate File Groups**
- Use `Glob` to find clusters of files that follow the same naming pattern (e.g., `*.controller.ts` always paired with `*.service.ts` and `*.module.ts`).
- Flag any directory where 3+ files share the same base name with different extensions or suffixes.

**Repeated Code Structures**
- Use `Grep` to find repeated import blocks, class/function signatures, or config patterns that appear in 5+ files.
- Look for copy-paste indicators: identical multi-line blocks across different files.

**Git History Patterns**
- Use `Bash` (read-only) to run:
  - `git log --oneline -50` — recent commit messages for recurring task types (e.g., "add migration", "update schema", "fix lint").
  - `git log --diff-filter=A --name-only --pretty=format: -50 | sort | head -30` — recently added files to spot scaffolding patterns.
  - `git log --all --oneline | head -100` — broader history scan for repeated workflows.
- Identify commit message patterns that suggest manual, repeatable processes.

**Manual Process Indicators**
- Use `Grep` to search for TODO/FIXME comments that reference manual steps.
- Look for shell scripts, Makefiles, or npm scripts that chain multiple commands together — these are candidates for skill automation.
- Check for checklist-style comments or documentation that describes step-by-step procedures.

### 4. Evaluate and Prioritize

Score each finding on:
- **Frequency**: How often does this pattern repeat? (daily > weekly > occasionally)
- **Effort saved**: How many manual steps or files does it involve each time?
- **Error risk**: Is this something people get wrong when doing it manually?
- **Scope**: Does it affect one project or could the skill be reused across repos?

### 5. Generate Output

Write the results to the output path (default: `skills-suggestions.md` in the project root). If the user provided an `[output-path]` argument, use that instead.

The output file must follow this structure:

```markdown

## Guidelines

- Generate **3-7 recommendations**, ordered by impact. Don't pad with low-value suggestions.
- Every recommendation must cite specific evidence from the codebase (file paths, pattern counts, git log entries).
- Draft SKILL.md skeletons should be functional starting points, not placeholders. Someone should be able to copy one into `~/.claude/skills/` and start using it with minor edits.
- If the codebase already has skills or automation in place, acknowledge what's covered and focus on gaps.
- If the codebase is small or has minimal repetition, say so honestly — don't invent problems. It's fine to output fewer than 3 recommendations.
- Use `Bash` only for read-only git commands. Do not modify the repository.
